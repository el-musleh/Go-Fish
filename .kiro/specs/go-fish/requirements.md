# Requirements Document

## Introduction

Go Fish is a web application that helps groups of people find a shared activity that fits everyone's preferences and availability. An inviter creates an event, shares a link with invitees, and after collecting responses, a Gemini AI agent analyzes preferences and availability to generate three activity options. The inviter picks the winning option, and all participants receive an email with the event details.

## Glossary

- **Inviter**: The user who creates an event and shares the invitation link with others
- **Invitee**: A user who receives an invitation link and responds with their preferences and availability
- **Event**: A planned group activity created by the Inviter, containing a title, description, and collected responses
- **Invitation_Link**: A unique, shareable URL that allows Invitees to respond to an Event
- **Taste_Benchmark**: A profile of user preferences built from a 10-question checkbox form, completed on first login
- **Decision_Agent**: The Gemini-powered AI component that analyzes responses and generates activity options
- **Response_Window**: The 24-hour period after Event creation during which Invitees can submit their responses
- **Activity_Option**: One of three suggested activities generated by the Decision_Agent based on collected responses
- **Auth_Service**: The authentication component supporting Google OAuth and email-based login
- **Backend**: The Node.js server application running inside a Docker container
- **Database**: The persistent data store running inside a Docker container

## Requirements

### Requirement 1: User Authentication

**User Story:** As a user, I want to log in with Google or email, so that I can access the application securely.

#### Acceptance Criteria

1. WHEN a user navigates to the application, THE Auth_Service SHALL present login options for Google OAuth and email-based authentication
2. WHEN a user selects Google OAuth, THE Auth_Service SHALL authenticate the user through the Google OAuth 2.0 flow and create a session
3. WHEN a user selects email-based login, THE Auth_Service SHALL authenticate the user through an email verification flow and create a session
4. IF authentication fails, THEN THE Auth_Service SHALL display a descriptive error message to the user
5. WHEN a user is authenticated, THE Auth_Service SHALL redirect the user to the appropriate page based on their role (Inviter dashboard or Invitee response form)

### Requirement 2: Taste Benchmark Creation

**User Story:** As a first-time Invitee, I want to complete a preference questionnaire, so that the system can understand my activity preferences.

#### Acceptance Criteria

1. WHEN an Invitee logs in for the first time, THE Backend SHALL present a 10-question checkbox form to create a Taste_Benchmark
2. THE Backend SHALL require the Invitee to complete the Taste_Benchmark form before accessing any Event response form
3. WHEN the Invitee submits the Taste_Benchmark form, THE Backend SHALL store the Taste_Benchmark associated with the Invitee's account
4. WHEN a returning Invitee logs in, THE Backend SHALL skip the Taste_Benchmark form and navigate directly to the Event response form
5. IF the Invitee submits an incomplete Taste_Benchmark form, THEN THE Backend SHALL display a validation error indicating which questions remain unanswered

### Requirement 3: Event Creation

**User Story:** As an Inviter, I want to create an event, so that I can organize a group activity.

#### Acceptance Criteria

1. WHEN an Inviter submits the event creation form, THE Backend SHALL create a new Event with a title and description
2. WHEN an Event is created, THE Backend SHALL start a 24-hour Response_Window for that Event
3. THE Backend SHALL associate the created Event with the authenticated Inviter
4. IF the Inviter submits an event creation form with missing required fields, THEN THE Backend SHALL display a validation error listing the missing fields

### Requirement 4: Invitation Link Generation

**User Story:** As an Inviter, I want to generate a shareable link for my event, so that I can invite others to participate.

#### Acceptance Criteria

1. WHEN an Inviter requests a link for an Event, THE Backend SHALL generate a unique Invitation_Link for that Event
2. THE Backend SHALL make the Invitation_Link copyable to the clipboard
3. WHEN an Invitee opens an Invitation_Link, THE Backend SHALL redirect the Invitee to the Auth_Service if not authenticated, and then to the Event response form
4. THE Backend SHALL ensure each Invitation_Link maps to exactly one Event

### Requirement 5: Invitee Response Submission

**User Story:** As an Invitee, I want to submit my available dates for an event, so that the group can find a time that works.

#### Acceptance Criteria

1. WHEN an authenticated Invitee accesses an Event response form, THE Backend SHALL display a date selection interface
2. WHEN an Invitee submits available dates, THE Backend SHALL store the response associated with the Invitee and the Event
3. WHILE the Response_Window is open, THE Backend SHALL accept responses from Invitees
4. IF an Invitee submits a response after the Response_Window has closed, THEN THE Backend SHALL display a message indicating the response period has ended
5. THE Backend SHALL allow each Invitee to submit only one response per Event

### Requirement 6: Activity Option Generation

**User Story:** As an Inviter, I want the system to suggest activities based on everyone's preferences, so that I can pick something the group will enjoy.

#### Acceptance Criteria

1. WHEN all Invitees have submitted responses for an Event, THE Decision_Agent SHALL analyze the responses and Taste_Benchmarks and generate three Activity_Options
2. WHEN the Response_Window expires for an Event, THE Decision_Agent SHALL analyze all received responses and Taste_Benchmarks and generate three Activity_Options, regardless of whether all Invitees have responded
3. THE Decision_Agent SHALL include a title, description, and suggested date for each Activity_Option
4. THE Decision_Agent SHALL rank the three Activity_Options by estimated group compatibility
5. IF fewer than two Invitees have responded when the Response_Window expires, THEN THE Backend SHALL notify the Inviter and allow the Inviter to extend the Response_Window or proceed with available responses

### Requirement 7: Activity Selection by Inviter

**User Story:** As an Inviter, I want to choose from the suggested activities, so that I can finalize the group event.

#### Acceptance Criteria

1. WHEN the Decision_Agent generates Activity_Options, THE Backend SHALL notify the Inviter and display the three Activity_Options
2. WHEN the Inviter selects an Activity_Option, THE Backend SHALL mark that Activity_Option as the chosen event
3. THE Backend SHALL allow the Inviter to select exactly one Activity_Option per Event

### Requirement 8: Event Notification Email

**User Story:** As a participant, I want to receive an email with the finalized event details, so that I know what was planned.

#### Acceptance Criteria

1. WHEN the Inviter selects an Activity_Option, THE Backend SHALL send an email to all Invitees who responded to the Event
2. THE Backend SHALL include the activity title, description, and date in the notification email
3. THE Backend SHALL send a confirmation email to the Inviter with the same event details
4. IF email delivery fails for an Invitee, THEN THE Backend SHALL retry delivery up to three times with a 5-minute interval between attempts

### Requirement 9: Containerized Infrastructure

**User Story:** As a developer, I want the backend and database to run in Docker containers, so that the application is portable and easy to deploy.

#### Acceptance Criteria

1. THE Backend SHALL run as a Node.js application inside a Docker container
2. THE Database SHALL run inside a separate Docker container
3. THE Backend SHALL connect to the Database through a Docker network
4. IF the Backend container fails to start, THEN THE Backend SHALL log a descriptive error message to the container output
5. IF the Database container is unreachable, THEN THE Backend SHALL retry the connection up to five times with a 3-second interval between attempts
